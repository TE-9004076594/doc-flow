## Context

企业正式文档自动化平台旨在解决企业文档生产中模板分散、重复填写、字段遗漏、格式错乱、审核无留痕等痛点。MVP 阶段的目标是建立**模板驱动 + 规则引擎 + 文档生成**的核心闭环，验证复杂模板保真与基础生成能力。

当前状态：全新项目，从零开始构建。技术栈选择优先考虑文档处理生态成熟度和团队开发效率。

### Constraints
- Word 模板格式保真度是核心竞争力，必须确保生成文档与模板样式高度一致
- 在线预览需接近 Word 最终效果
- 批量生成需要异步处理以避免请求超时
- MVP 范围优先，架构需为 V1（审核流、知识库、API 集成）预留扩展点

### Stakeholders
- 产品团队：定义功能和用户流程
- 研发团队：后端（文档处理、规则引擎、API）、前端（模板配置、预览、编辑）
- 测试团队：模板生成成功率、格式保真率验证
- 实施团队：企业客户部署与使用

## Goals / Non-Goals

**Goals:**
- 实现 Word 模板上传、解析、变量提取与替换
- 实现条件块/循环块的结构化解析与渲染
- 实现基础校验规则引擎
- 实现单份和基础批量文档生成
- 实现在线分页预览与轻量编辑
- 实现 Word/PDF 格式导出
- 实现基于角色的基础权限控制
- 架构上为 V1（审核流、知识库、API 集成）预留扩展能力

**Non-Goals:**
- Excel 模板生成（V2 范围）
- 高级规则引擎（V2 范围）
- 多级审核流与审批流程（V1 范围）
- 企业知识库接入与历史文档引用（V1 范围）
- AI 模板推荐与内容补全（V2 范围）
- 系统集成 API（V1 范围）
- 实时多人协同编辑（V2 范围）
- 行业解决方案包（V2 范围）

## Decisions

### D1：架构分层 — 前后端分离 + 文档处理微服务

**决策**：采用前后端分离架构，前端为 React SPA，后端为 RESTful API 服务，文档的模板解析与生成独立为 Document Engine 服务。

**理由**：
- 前后端分离有利于团队并行开发和独立部署
- 文档处理（模板解析、渲染、导出）是 CPU 密集型操作，独立服务可水平扩展且不影响 API 响应
- MVP 阶段可合并部署，V1 阶段可拆分

**备选方案**：
- 单体架构：初期开发速度快，但不利于扩展和隔离文档处理负载
- 全栈框架（Next.js/Nuxt）：SSR 对文档预览帮助有限，且文档服务解耦更灵活

### D2：技术栈选型

| 层级 | 技术 | 理由 |
|------|------|------|
| 前端 | React + TypeScript | 生态成熟，组件化开发，TypeScript 提升代码质量 |
| 后端 | Python (FastAPI) | 文档处理生态（python-docx）成熟，FastAPI 异步支持好，代码简洁 |
| 文档引擎 | Python (python-docx, lxml) | python-docx 是 .docx 解析最成熟的库，可直接操作 OOXML |
| 数据库 | PostgreSQL | 支持 JSON 字段存储模板结构，扩展性强 |
| 文件存储 | 本地文件系统（S3 接口预留） | MVP 阶段本地存储即可，预留 S3 接口便于后续迁移 |
| 任务队列 | Celery + Redis | 成熟可靠，支持异步批量任务 |
| PDF 转换 | LibreOffice headless | 成熟稳定的 docx → PDF 转换方案 |

### D3：模板解析策略 — 占位符标记 + OOXML 结构操作

**决策**：模板内使用 `{{变量名}}` 作为占位符标记，通过解析 OOXML XML 树结构定位占位符；条件块使用 `{{#if:条件}}...{{/if}}`，循环块使用 `{{#each:列表}}...{{/each}}` 语法标记。

**理由**：
- 用户可以在 Word 中直接编辑模板，不依赖专用编辑器
- OOXML 结构操作可精准保持原始样式（字体、颜色、段落格式、表格等）
- 标记语法清晰，易于学习和调试

**备选方案**：
- 使用 Word 内容控件（Content Control）：需要商业库或复杂实现，用户编辑门槛高
- HTML 转 Word：格式保真度低

### D4：预览方案 — 基于 OOXML 渲染为 HTML

**决策**：后端将 OOXML 解析为精简 HTML，前端使用 `react-doc-viewer` 或自定义渲染器展示。

**理由**：
- Web 端无法直接展示 .docx，必须转为 Web 可渲染格式
- 后端直接解析 OOXML → HTML 可确保与最终 Word 输出的一致性
- 避免依赖第三方预览服务（安全可控）

**备选方案**：
- 嵌入 Microsoft Office Online / WPS 在线预览：依赖外部服务，控制力弱
- 转换为 PDF 后使用 PDF.js 预览：丢失可编辑能力

### D5：权限模型 — RBAC 基础权限

**决策**：采用基于角色的访问控制（RBAC），MVP 阶段定义三个角色：系统管理员、模板管理员、普通用户。权限粒度到模板级别。

**理由**：
- RBAC 是业界标准权限模型，易于理解和实现
- 三个角色覆盖 MVP 核心场景
- V1 可扩展为更细粒度的权限体系

## Risks / Trade-offs

- **[风险] Word 复杂格式还原难度高**：表格合并单元格、嵌套列表、图文混排等复杂结构解析存在边界情况。→ **缓解**：MVP 优先覆盖 80% 常见场景，复杂场景逐步迭代优化；建立格式保真率测试套件
- **[风险] 在线预览与最终 Word 效果不一致**：HTML 渲染与 Word 渲染引擎存在差异。→ **缓解**：预览明确标注"近似效果"，提供下载查看最终文档的明确入口
- **[风险] 模板标记语法侵入 Word 编辑体验**：`{{}}` 标记在 Word 中可能影响编辑。→ **缓解**：提供模板语法高亮预览工具，支持模板语法校验
- **[风险] 批量任务规模预估不足**：MVP 阶段批量生成可能面临性能瓶颈。→ **缓解**：采用异步队列处理，支持并发任务控制，预留任务优先级字段
- **[风险] 团队对 python-docx 操作 OOXML 的复杂度**：OOXML 是复杂的 XML 规范。→ **缓解**：建立文档引擎核心抽象层，封装常见操作；编写集成测试覆盖关键模板场景
- **[权衡] Python + React 异构技术栈**：前后端语言不同，增加团队沟通和人才招聘成本。→ **补偿**：后端通过清晰的 REST API 接口与前端解耦，接口以 OpenAPI 规范定义
