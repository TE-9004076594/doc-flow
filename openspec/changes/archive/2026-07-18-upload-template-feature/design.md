## Context

模板上传功能的后端 API（`POST /api/templates`）已在 MVP 中实现，支持接收 .docx 文件和模板元数据。但前端目前仅有占位按钮，无实际上传交互。本次变更聚焦于**完善前端上传流程**，覆盖从文件选择到模板创建成功的完整体验。

### 当前状态
- 已有后端 `POST /api/templates` 端点（multipart/form-data：file, name, description, category, tags）
- 前端模板列表页 (`Templates.tsx`) 有"上传模板"按钮但无交互逻辑
- 模板详情页 (`TemplateDetail.tsx`) 已有变量编辑、标签展示等功能

### 约束
- 仅修改前端代码，不修改后端 API
- 复用现有 API 参数格式
- 保持与现有 Ant Design 风格的 CSS 变量一致

## Goals / Non-Goals

**Goals:**
- 点击"上传模板"弹出模态框，支持拖拽上传和文件选择
- 上传前验证：仅 .docx 格式、不超过 50MB
- 文件选中后显示上传进度
- 上传成功后弹出元数据配置表单（名称、分类、标签、描述）
- 提交后调用 API 创建模板
- 创建成功后显示解析结果摘要（变量数量、逻辑块数量），并提供跳转到详情页的入口
- 上传过程中错误以醒目方式展示

**Non-Goals:**
- 模板高级编辑（已在模板详情页实现）
- 批量上传（后续版本）
- 模板解析结构的可视化编辑（已在模板详情页实现）

## Decisions

### D1：上传方式 — 模态框（Modal）而非独立页面

**决策**：采用模态框承载上传流程，分两步：Step 1 文件选择，Step 2 元数据配置。

**理由**：
- 上传是一个临时操作，模态框不会打断用户在模板列表页的上下文
- 两步式流程让用户专注于当前步骤
- 实现简单，不需要新的路由

**备选方案**：
- 独立上传页面：需要新路由，页面切换增加操作成本

### D2：拖拽上传 — 使用原生 HTML5 Drag & Drop API

**决策**：使用原生 HTML5 Drag & Drop API 实现文件拖拽区域，而非第三方库。

**理由**：
- 功能需求简单（仅 .docx 文件拖拽），无需引入 react-dropzone 等外部依赖
- 控制更精细，代码无冗余
- 减小打包体积

### D3：上传后解析结果展示 — 调用现有 `/api/templates` 响应

**决策**：利用后端 `POST /api/templates` 返回的模板数据（id, name, status, current_version），前端自行解析展示。

**理由**：
- 后端已返回所有必要信息
- 无需额外 API 调用
- 首版展示变量数量（从模板详情 API 获取）

## Risks / Trade-offs

- **[风险] 大文件上传期间 UI 卡顿**：50MB 文件上传需要时间。→ **缓解**：添加上传进度条（axios 的 onUploadProgress），过程中禁用操作按钮
- **[经验] Python 后端文件读取的同步阻塞**：FastAPI 的 UploadFile.read() 是同步调用，大文件上传会阻塞事件循环。→ **缓解**：MVP 阶段可接受，V1 考虑使用流式上传
